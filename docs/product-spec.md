## 아키텍처 원칙 (2026-07-12 — Inline AI 비교 분석에서 확정)

경쟁(Inline AI) = 설치된 한글을 원격 조종하는 로컬 RPA 에이전트. **우리 = 에디터 그 자체(웹)** —
조종할 외부 프로그램이 없으므로 RPA 계층이 통째로 불필요(구조적 우위). 로컬 에이전트 노선은
별도 프로젝트(junha-ai)의 영역이며 이 저장소에 섞지 않는다.
1. **로컬 퍼스트**: 문서는 브라우저 안에서 처리(현재 서버 저장 없음). 저장(P2-9) 도입 후에도
   "AI엔 문서 전체가 아니라 필요한 맥락만" 원칙 유지.
2. **최소 맥락 전송**: AI 호출마다 전송 범위를 명시적으로 정의(선택 텍스트·대상 표만) —
   문서 전체 덤프 금지. 전송 내용 목록은 P2-8 데이터 처리 고지와 1:1 대응.
   **예외(2026-07-12)**: "문서 전체 검토"처럼 전체가 본질인 기능은 전체 전송을 허용하되,
   **기본 동작 금지 + 전송 내용을 화면에 명시("글상자 N개·M자") + 사용자 동의** 3조건을
   반드시 갖춘다(원칙의 의도=몰래 덤프 금지를 지킴). 이 동의 화면이 곧 P1-8의 첫 구현.
3. **승인 루프(HIL)**: AI의 문서 변경은 계획/전후 비교 → 사용자 승인 → 단일 스냅샷 적용
   (Ctrl+Z 일괄 취소). 이미 구현된 이 패턴이 **모든 신규 AI 기능의 기본**이다(다단계 작업 포함).

  rhwp-adoption.md에 3줄 기록. (AI가 못 하는 유일한 검증 — 사용자 루틴, 회당 ~10분)

### P1 — AI 심화 (차별점 강화)

  findings 리스트(위치 맥락 "표 N행 M열")→개별 적용=단일 스냅샷·Ctrl+Z 취소. 표현·오탈자

   기존 AI 문서 생성 파이프라인으로 A4 회의록 배치. 전략 = STT는 빌리고(①), 구조화는

   비교는 v2 — v1을 실회의 1건 도그푸딩 후 tech-choices 절차로.

## AI 조언 의무 — 제안의 기준 (운영 규칙은 playbooks/agent-protocol.md §4 기능 조언 의무)

- **기능 제안**: 이 로드맵과 "차별점 3종"에 부합하는 것만. 제안 형식 = "무엇을(1줄) + 로드맵
  어디에 끼우나(P0~P3) + 왜 지금(1줄)". 로드맵에 없는 신규 아이디어는 P 배정을 제안하고
  사용자 승인 후 이 문서에 추가. **신규 아이디어의 기본 착지는 P3(백로그)** — P0 진입은 기존
  P0 항목 1개 완료 또는 명시적 제외와 맞교환(1-in-1-out). 다음 착수 후보 제안은 P0 최상단
  미완 항목을 기본으로.
- **라이브러리 추천**: 절차·비교 기준의 정본 = docs/tech-choices.md "라이브러리 채택 절차" 절
  (여기 재서술 금지 — H4).
- **구현 가이드**: 채택 시 단계별 계획(파일 단위 + 검증 방법 + 완료 게이트)을 코드 작성 전에 제시.
