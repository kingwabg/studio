# 규칙 변경 이력 — 초보자용 (최신이 맨 위)

이 파일은 **규칙 문서(CLAUDE.md · docs/playbooks/* · .claude/commands/*)가 바뀔 때마다**
한 항목씩 쌓입니다. 형식: 날짜 → 무엇이 → 왜 → 쉽게 말하면. 코드 변경은 여기 안 적습니다
(그건 git log). **규칙 문서를 고친 사람(AI 포함)은 반드시 맨 위에 항목을 추가할 것.**

## 규칙 문서 지도 (뭐가 어디 있나 — 1줄씩)

| 파일 | 역할 |
|---|---|
| `CLAUDE.md` (저장소 루트) | 헌법 — 프로젝트 구조·불변식·금지·검증된 지식. AI가 매 세션 자동으로 읽음 |
| `docs/playbooks/agent-protocol.md` | AI 운영 수칙 — 세션 시작 순서, 작업 루프, 하지 말 것, 막힐 때 |
| `docs/playbooks/traps.md` | 함정 사전 — "이 증상이면 이 원인" 모음 (실측 확정만) |
| `docs/playbooks/verify.md` | 검증 규칙 — 무엇을 바꾸면 어떤 테스트를 돌려야 하는지 + 완료 보고 양식 |
| `docs/playbooks/browser-drive.md` | rhwp 에디터를 자동화로 조작·측정하는 레시피 |
| `docs/playbooks/debugging.md` | 처음 보는 버그를 잡는 절차서 ("창의적 디버깅"의 체크리스트판) |
| `.claude/commands/boot.md, done.md` | 슬래시 명령 — `/boot`(세션 시작), `/done`(완료 검사) |
| `docs/rhwp-adoption.md` | 규칙 아님 — 작업 일지(역사). 함정의 상세 서사가 여기 있음 |

## 2026-07-12

### ➕ 작업 티어 라우팅 (S/M/L) — 작은 수정에 규칙 전체를 적용하지 않기
- **왜**: 준하 님 지적 — "버튼 모양 수정마저 규칙을 하나하나 파악해야 하면 별로다."
- **쉽게 말하면**: 규칙은 체크리스트가 아니라 상황별 걸림선. 이제 작업 시작 때 크기만 판정 —
  버튼 수정 같은 S(경량)는 규칙 딱 3개(팔레트 준수·전/후 1개·자동 훅)로 끝. 기능 추가는 M(표준
  루프), 코어·구조는 L(상의 필수). 애매하면 M으로. /done도 S면 간이 게이트만.

### ➕ AI 기능 조언 시스템 — product-spec(기능 명세 정본) + tech-choices(기술 선택 대장)
- **왜**: 준하 님 지시 — "기능 명세를 문서로 두고, AI가 수시로 확인해서 기능 제안·라이브러리
  추천·구현 가이드를 하는 시스템" + "순수 자체구현 기능들도 라이브러리 필요한지 점검."
- **쉽게 말하면**: 이제 AI는 일 끝날 때마다 로드맵(docs/product-spec.md)을 보고 "다음엔 이거
  하면 좋겠다"를 제안하고, 새 기능 전엔 기술 대장(docs/tech-choices.md)을 보고 "이 라이브러리
  쓸까요/직접 만들까요"를 비교해서 물어봅니다. (운영 규칙: agent-protocol §4)
- 함께: 자체 구현 30건 전수 심사 완료 — 즉시 교체 1건(SheetJS 보안), 트리거 대기 4건,
  유지 25건(근거 기록). 부수 코드 결함 4건 발견(대장의 "부수 발견" 절).

### ➕ 낯선 버그 디버깅 절차서 신설 — `debugging.md` (커밋 0a34380)
- **왜**: "낮은 모델은 창의적 디버깅을 못 하니 상위 모델로 넘겨라"는 이전 규칙을 준하 님이 반박
  — "그 창의적 디버깅을 0.1%까지 절차로 적어라."
- **쉽게 말하면**: 버그를 만나면 감이 아니라 이 체크리스트대로: 재현 고정 → 증상을 숫자로 →
  원인 후보를 표(장부)로 관리 → 싼 실험부터 → 막히면 8가지 확장 기법. 실제 있었던 버그 2개로
  "절차만 따라가도 정답에 도달"하는지 시뮬레이션까지 통과.
- 함께: agent-protocol의 "막히면 에스컬레이션" → "먼저 debugging.md 소진, 넘길 땐 장부 첨부"로 수정.
