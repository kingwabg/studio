# 에이전트 운영 프로토콜 — 모든 모델 공통 (낮은 모델일수록 이 문서가 안전벨트)

목적: 어떤 모델(Haiku/Sonnet/Opus/Fable)이 세션을 잡아도 같은 품질·같은 안전선으로 작업하게 한다.
토큰 절약이 설계 원칙: **항상 읽는 건 CLAUDE.md뿐, 나머지는 작업이 닿을 때만 읽는다.**

## 0. 세션 부팅 (순서 고정)

1. CLAUDE.md (자동 로드) — 헌법: 불변식·금지·구조.
2. 이 파일 (agent-protocol.md).
3. **여기서 멈춤.** 다른 플레이북·로그는 작업이 해당할 때만:
   - 함정 의심/디버깅 → `docs/playbooks/traps.md`를 증상 키워드로 **grep** (통독 금지)
   - 검증·완료 보고 → `docs/playbooks/verify.md`
   - rhwp UI 실구동 → `docs/playbooks/browser-drive.md`
   - 역사·상세 맥락 필요 → `docs/rhwp-adoption.md`도 **grep으로 절만** (통독은 토큰 낭비)

## 1. 작업 루프 (매 작업)

1. **함정 먼저**: 만질 영역의 키워드로 traps.md grep — 같은 증상/영역 항목이 있으면 그 해법 준수.
2. **최소 diff**: 통짜 재작성 금지. rhwp-studio 수정엔 반드시 `[캔버스 한컴 포크]` 주석.
   새 기능은 새 파일 우선(성장 규칙), 기존 파일엔 배선만.
3. **검증**: verify.md 매트릭스대로 실행. UI면 browser-drive.md로 실구동 실측.
4. **보고**: 완료 게이트 양식 + (UI면) 이전/이후 실측. 미검증은 미검증이라고 쓴다.
5. **기록**: 새 함정을 실측으로 확정했으면 traps.md에 3줄 추가(서사는 rhwp-adoption.md).
6. **규칙 변경 이력** (2026-07-12 신설): 규칙 문서(CLAUDE.md·플레이북·명령)를 바꿨으면
   `docs/playbooks/CHANGELOG.md` **맨 위에 항목 추가가 필수** — 형식: 날짜 / ➕신설·✏수정 /
   무엇 / 왜 / "쉽게 말하면"(초보자용 한 줄). 가능하면 고친 지점에도 (YYYY-MM-DD) 표기.

## 2. 하지 말 것 (약한 모델의 전형적 사고 지점)

- 검증 없이 "완료/고쳤다" 선언. 코드만 읽고 "될 것"이라 선언.
- rhwp **엔진·문서 모델 수정** (입력 해석 레이어만 허용 — CLAUDE.md 대원칙).
- 별도 JSON 캔버스 모델을 rhwp에 이식 (진실 2개 금지).
- 파일 통짜 재작성·대량 삭제 (사용자 확인 없이).
- 커밋·푸시를 지시/승인 없이. `[size-override]` 자가 판정. 게이트 실패를 우회(--no-verify 등).
- 비밀값(.env.local·키)을 커밋/예시 파일에. 로컬 동결 클론에서 pull/push/reset.
- `npm test` 생략한 rhwp 포크 (소스 계약 테스트가 조용히 깨진다 — 실사고 있음).
- 실패한 검증을 "타이밍 문제일 것"으로 덮기.

## 3. 낯선 문제(새 버그) — 에스컬레이션 전에 디버깅 프로토콜

traps.md에 없는 새 증상은 **`docs/playbooks/debugging.md` 절차를 먼저 수행**한다 — 재현 고정 →
증상 정밀 기술 → 인과 사슬 → 계측 이분 → 가설 장부 → 판별 실험 → (막히면) 가설 확장 기법 a~h.
"창의적 디버깅"은 그 문서에 절차로 적혀 있다 — 낮은 모델도 따라가면 대부분 도달한다.

**에스컬레이션 (마지막 수단)** — 다음이면 멈추고 보고 + 상위 모델 전환 제안:
- debugging.md 7단계 a~h를 **전부 소진**하고도 생존 가설 0. (이때 반드시 **가설 장부 전체**를
  넘긴다 — 다음 모델이 0에서 재시작하지 않게.)
- CLAUDE.md **불변식/검증된 매핑과 충돌**하는 변경이 필요해 보일 때.
- 크기 게이트·테스트가 막는데 우회 없이는 진행 불가일 때.
- 되돌리기 어려운 결정 (스키마·저장 형식·공개 API·대량 마이그레이션).

## 4. 보고 형식 (사용자 상시 지시)

- 매 턴 첫머리 배지 1줄: `▸ <모델> · <작업밀도 낮음/중간/높음/최대> — 이유`.
- 결론 먼저, 증거(실측)와 함께. 다음 할 일 나열보다 실제 실행.
- UI/동작 변경 = 이전/이후 비교(+실측값). 위젯 가능 환경이면 위젯, CLI면 표.

## 5. 토큰 절약 수칙

- 파일은 필요한 부분만 읽기(grep→offset 읽기). 큰 로그(rhwp-adoption.md)는 항상 grep.
- 같은 정보 재확인 금지 — 이 세션에서 이미 실측한 값은 재측정하지 않는다.
- 탐색이 3파일을 넘을 것 같으면 서브에이전트(탐색용 저비용 모델)에 위임하고 결론만 받는다.
- 무거운 플레이북/로그 통독 금지 — 이 문서의 부팅 순서가 곧 로딩 예산이다.
