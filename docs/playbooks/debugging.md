# 낯선 문제 디버깅 프로토콜 — "창의적 디버깅"의 완전 명시화

**철학**: 디버깅은 영감이 아니라 증거 수집 절차다. 상위 모델의 "창의성"의 실체는
①가설 공간을 넓히는 기법 목록 ②반증 규율 ③실험 설계 패턴 — 전부 아래에 절차로 적었다.
모든 기법에 이 저장소의 **실사례**를 증거로 달았다 (전부 실제로 이 방법으로 풀린 것).
낮은 모델이 이 문서로 강해지는 이유: 판단을 절차가 대신한다. 상위 모델도 따라야 하는 이유:
규율이 직감의 오류(성공 신호 맹신·가정 무검증·다중 원인 간과)를 막는다.

**사용법**: −1단계부터 순서대로. 각 단계는 건너뛰지 말 것 — 건너뛰고 싶은 단계가 보통 답이 있는
곳이다. (부록 A는 단계 생략용이 아니라 5단계 장부의 초기 가설을 채우는 힌트다.)

---

## −1단계. 선행 grep (이미 아는 함정인가?)

`docs/playbooks/traps.md`를 증상 키워드(예: "ok:true", "불변", "인라인", "width", "0")로 grep한다.
적중하면 그 3줄이 답이고 이 문서는 불필요. **적중 없음을 확인한 뒤에만** 0단계로 진입.

## 0단계. 분류 — 첫 질문 2개 (30초, 이후 전략이 갈림)

**Q1. 되던 게 안 되나(회귀), 처음부터 안 되나(신규)?**
- 회귀 → 시간축 이분이 코드 수사보다 싸다: `git log --oneline -- <파일>`로 용의 커밋 나열.
  **5개 이하면** 최신부터 diff를 눈으로, **6개 이상이면** `git bisect`(E0 재현 스크립트를 판정기로).
  코드 말고도: 의존성 버전·환경·데이터가 "바뀐 것"일 수 있다.
- 신규 → 최소 작동 사례에서 쌓아 올리기: 가장 단순한 형태(새 문서·표 1개·글자 1개)로 먼저
  성공시키고, 실패 케이스와의 차이를 한 겹씩 추가하며 어느 겹에서 깨지는지 찾는다.

**Q2. 결정적인가, 간헐적인가?**
- **동일 조건(리로드 여부·캐시 상태·입력 순서)을 고정**하고 5회 반복, 재현율 기록(5/5? 2/5?).
  0/5이면 "재현 없음" — 1단계의 관찰 수집 모드로.
- 간헐이면 원인 후보가 다르다: 타이밍(부팅 직후·RAF·setTimeout)·캐시(HMR 잔재·stale rect)·순서
  의존. → 실사례: 리로드 직후 첫 더블클릭만 무시(부팅 타이밍) — 재현율 기록 덕에 "코드 버그
  아님·재시도 우회"로 정확히 분류.
- 간헐 문제를 "한 번 됐으니 고쳐졌다"로 판정하는 것이 최악의 오판 (종료 기준은 8단계에 별도).

## 1단계. 재현 고정 (모든 것의 전제 — 이것 없이 진행 금지)

- **온디맨드 재현 E0**: 명령/스크립트 하나로 언제든 버그를 일으킬 수 있게 만들고 E0으로 번호를
  붙인다.
- **실험 격리**: persist 상태(자동저장·IndexedDB·localStorage·HMR 잔재)는 이전 실험의 실패
  산물을 다음 실험에 주입한다. E0에 **초기 상태 리셋 단계를 포함**하라 — 리셋 없는 재현은
  재현이 아니다. → 실사례: 자동저장(rhwpStudioAutosave)이 깨진 표를 저장해 "새 문서"가 오염된
  채 부팅.
- **최소화(축 순서 고정)**: ①요소 개수 ②조작 단계 수 ③콘텐츠 길이 순으로, 각 축에서 절반을
  제거하고 E0을 재실행한다. 버그가 남으면(판정 = 2단계의 실측 수치가 동일 재현 — 비슷한 증상
  아님) 그 절반을 버린 채 계속, 사라지면 방금 제거한 절반을 되살려 그 안을 다시 반으로.
  종료 조건: 어느 하나를 더 제거해도 재현이 사라지는 상태(1-minimal).
- **재현이 안 되면** 디버깅이 아니라 관찰 수집 단계다: 3단계 인과 사슬을 먼저 그리고 각
  경계(우리↔라이브러리, JS↔WASM)에 로그 1개씩 심은 뒤 원 증상의 조작을 **10회** 반복. 0회면
  사용자에게 재현 조건(환경·직전 조작·데이터)을 질문하고 상한(자연 재현 1회 또는 1일)을 정해
  대기 — 상한 도달 시 단서가 있으면 0-Q2로, 없으면 9단계 형식으로 기록만 남기고 종료.
  **재현 없이 수정 커밋 금지.**

## 2단계. 증상 정밀 기술 — 관찰과 해석을 분리 (해석 금지 구간)

- **에러 원문을 그대로 복사** (요약·의역 금지). **발신 층 식별 절차**: ①스택 최상단 프레임의
  파일 경로(src/=우리, node_modules/<pkg>/=그 라이브러리, 경로 없음=브라우저 내장) ②스택이
  없으면 에러 문구의 고유한 6~10단어를 따옴표째 grep — src/ 히트=우리 층, node_modules
  히트=라이브러리, 무히트=원격 API/브라우저(네트워크 응답 본문 확인). 층이 곧 수색 범위다.
  → 실사례: AI 401 메시지가 "x-api-key required" → (키 주입 후) "CORS requests must set …"로
  **바뀜** = 한 층 통과의 증거. 새 에러는 후퇴가 아니라 전진이다.
- **예상된 실패는 도달 증거다**: 자격/자원 없이도 파이프라인을 검증할 수 있다 — "올바른 층의
  올바른 형식의 에러"(원격 API의 진짜 401 JSON, 404·프록시 HTML 아님)가 오면 전달은 성공.
  → 실사례: 키 없이 프록시 검증(실제 401=전달 성공), 크레딧 부족 400=코드 아닌 과금 이슈로 분류.
- **"안 된다" 서술 금지** — 어디까지 되고 어디부터 다른가를 수치로: "표가 안 움직임"(나쁨) vs
  "ok:true 반환, 렌더 y=125.4mm 불변, 목표 95"(좋음). 기대값·실측값을 숫자로 나란히.
- **차이를 아는 상수와 대조하라** (상수 풀 정본 = traps.md "단위 혼동" + CLAUDE.md "검증 완료된
  HWPX 매핑" + 해당 코드 constants 선언부 — 열어서 후보를 만들고 차이값의 정수배·근사 일치를
  찾는다). → 실사례: 어긋난 44.7mm ≈ 5행 표 높이 → "다른 표가 흐름을 밀었다" 즉시 도출.
  잔차 1mm = 283HWPUNIT = 표 바깥여백 → 보정값 확정.
- **증상을 적었으면 부록 A 표를 즉시 대조한다.** 패턴이 일치하면 그 행의 기법 순서를 5단계
  장부의 초기 가설 H1~H3으로 등록하고 시작한다 — 부록을 문서 끝에서 발견하지 말 것.

## 3단계. 인과 사슬 그리기 (파일에 — 머리 아님)

버그는 "입력→효과" 사슬의 끊어진 링크 하나다. 사슬을 명시적으로 적는다:

```
예: keydown → textarea 리스너 → onKeyDown 분기 → wasm.deleteText → 문서 상태 → 재조판 → 렌더
예: MINIMAX_API_KEY → loadEnv → vite proxy configure → proxyReq 헤더 → api.minimax.io → 응답
예(CSS): 클래스 토글 → classList 반영 → 선택자 매칭 → 캐스케이드(인라인>시트) →
        레이아웃 제약(flex min-width·min-content) → computed width → paint
```

적고 나면 질문이 자동으로 나온다: "신호가 어느 링크까지 도달하는가?" — 4단계.

## 4단계. 계측 이분 탐색

- 사슬의 **중간에 프로브 1개**(로그·상태 되읽기) → 신호 도달했으면 뒤쪽 절반, 아니면 앞쪽 절반.
  끊어진 링크가 log₂ 횟수에 잡힌다. → 실사례: `console.log(keyLoaded, len)` 1줄로 수색 범위가
  '헤더 주입 이후'로 절반 축소.
- **프로브는 경계에 우선**: 우리 코드↔라이브러리, 브라우저↔서버, JS↔WASM. 버그의 다수는 경계의
  계약 오해다.
- **계측기 캘리브레이션** — 다음 중 하나면 측정 도구부터 의심한다: ①수정을 가했는데 측정값이
  한 자리도 안 변함 ②서로 다른 두 측정 수단이 다른 값을 줌 ③측정값이 **전부 0·전부 동일**
  (거의 항상 버그가 아니라 도구/초기화 결함). 도구를 값이 알려진 정상 대상에 적용해 예상값이
  나오는지 먼저 확인. → 실사례: 자동화 브라우저 rect 전부 0(뷰포트 0 — resize_window로 해결) /
  변경 노드의 stale rect(프로브 div는 정상) / 로드 직후 셀 bbox 높이 전부 0(reflow 미호출).
- **관찰·구동 수단의 사다리**: UI 자동화가 막혀도 검증을 포기하지 말고 한 단씩 내려간다 —
  ①실사용자 조작 ②실이벤트 디스패치(단 **리스너가 등록된 요소에 정확히**: mousemove는 canvas·
  keydown은 textarea — window/document에 쏘면 안 닿는 리스너가 있다) ③스로틀 내부 핸들러 직접
  호출(RAF 우회) ④엔진 API 실측(wasm bbox·getTextInCell) ⑤전부 불가면 "미검증·사용자 확인
  필요"로 정직 종결. 내려갈수록 커버리지가 좁아지므로 **어느 단으로 검증했는지 기록**한다.
- **종료 조건**: 끊어진 링크 1개 특정. 끊긴 이유까지 자명하면 8단계로 직행, "어디가 끊겼는지는
  아는데 왜인지 모르면" 그 링크를 대상으로 5단계 장부를 연다.

## 5단계. 가설 장부 — 파일로 외부화 (머리로 들지 말 것)

**스크래치패드에 `debug-<증상 두세 단어>.md` 파일을 만들고** 아래 표를 기록, 실험마다 갱신한다
(응답 텍스트에만 쓰면 턴이 넘어갈 때 유실된다). ※ 5단계(장부)와 6단계(실험 패턴)는 선형이
아니라 **루프**다: 가설 등록 → 6단계 목록에서 가장 싼 실험 선택 → 결과 기록 → 판정 → 반복.

```
| # | 가설 | 반증 가능한 예측 (H면 측정 M이 X여야) | 실험 | 결과 | 판정 |
|---|------|----------------------------------------|------|------|------|
| H1 | 생성 순서가 앞 표를 민다 | 전부 생성 후 이동하면 전부 정위치 | E3: 2-phase | 2/3만 정위치 | 부분 기각 |
| H2 | 위쪽 이동이 클램프된다 | 아래 이동은 될 것 | E4: +20mm 아래 | 아래도 불변! | 기각 |
| H3 | 오프셋 자체가 무시된다 | 속성 되읽기에 offset 없을 것 | E5: getTableProperties | vertOffset=-40235 있음 | 기각 → 신규 관찰 |
```
(↑ 실제 표 이동 미스터리의 장부. H3 기각이 만든 신규 관찰 "오프셋은 쌓이는데 렌더만 불변"이
 vertRelTo:Para + restrictInPage 클램프라는 진짜 원인으로 직행시켰다.)

**장부 규칙**:
- 예측 없는 가설 등록 금지 — "~인 것 같다"는 "~라면 측정 M이 X일 것"으로 바꿔야 등록 가능.
- **실험당 변수 1개.** 실험은 E1, E2… 번호 + 결과를 기록(반복 낭비 방지).
- **판정 어휘는 {채택, 기각, 부분 기각, 보류} 고정.**
  채택 = 예측 적중 + 되읽기로 효과 확인 + 관찰된 **전 증상** 설명 가능 → 8단계로.
  부분 기각 = 살아남은 부분을 새 가설로 분리 등록. 생존 가설 0 → 7단계.
- 같은 가설을 새 정보 없이 재시도 금지. 2회 실패한 접근은 폐기 — **접근을 바꾼다 = 7단계 a~h
  에서 아직 안 쓴 기법을 위에서부터 선택**해 다음 실험을 설계한다. 같은 API에 파라미터만 바꿔
  재시도하는 것은 접근 변경이 아니다(6단계 우회처럼 **메커니즘**이 달라야 함).
- **기각도 성과다** — 기각이 만든 "이상한 잔여 관찰"이 다음 가설의 씨앗이다.
- **정체 체크포인트**: 최근 실험 3개가 새 관찰을 하나도 못 만들었으면, 생존 가설이 남아 있어도
  7단계로 강제 진입한다.

## 6단계. 판별 실험 설계 패턴 (가설을 가장 싸게 죽이는 법)

- **되읽기(read-back)**: 쓰기 API 호출 뒤 반드시 상태를 get으로 되읽는다. **성공 연극 불신
  목록**: 반환 ok / HTTP 200 / 콘솔 무에러 / 안 돌린 테스트의 초록불 / **mock 경계가 실계약과
  다른 자체 하네스의 초록불** (→ 실사례: 표 스케일 초판이 mock 하네스 13/13 통과, 실제
  드래그에선 열 재분배만 — 종결 조건은 실경로 구동). 되읽기/우회 API가 없어 보이면 단정 금지 —
  브리지 소스(wasm-bridge.ts, 320+ 메서드)를 grep.
- **선읽기(read-first)**: 쓰기 API의 파라미터 키가 불확실하면 대응 read API의 반환 키를 먼저
  읽는다 — 틀린 키는 에러 없이 조용히 무시되어 "했는데 안 됨"을 만든다. → 실사례: 문단 정렬
  키는 align이 아니라 **alignment**(getCellParaPropertiesAt로 확인 후 기록).
- **차등 실험(differential)**: 되는 개체와 안 되는 개체를 나란히 두고 차이 집합을 좁힌다.
  → 실사례: 새(fresh) 2×2 표는 이동됨 vs 기존 6×4는 불변 → 차이는 "앵커의 흐름 위치"로 확정.
- **우회 실험(bypass)**: 같은 목표를 다른 메커니즘으로. 성공하면 원인 격리와 해법을 동시에
  얻는다. 대체 메커니즘 후보의 발굴은 **7c의 검색 레시피를 선참조**(근거 없는 API 추측 금지).
  → 실사례: moveTableOffset(상대) 대신 setTableProperties(Paper 절대) — 그대로 해법.
- **주입 실험(injection)**: 의심 값을 직접 주입해 하류 검증. 주입값은 **우연히 맞을 수 없는
  서명값**으로. → 실사례: 정답 API 판별 불가 → 콘솔에서 후보들에 width=모델폭×1.4 주입·렌더
  실측 — setCellProperties만 전 열 정확히 ×1.40 (계약 규명과 해법 동시 획득).

## 7단계. 막혔을 때 — 가설 공간 넓히기 ("창의성"의 기법 목록)

**진입 조건(막힘의 정의)**: 장부의 생존 가설 0, **또는** 생존 가설이 반증 가능한 예측을 만들지
못함, **또는** 정체 체크포인트 발동.
**예산**: a~h **최대 1순회**. 각 기법은 실험 2회 또는 30분 소진 시 "수행 완료"로 다음으로.
순회 중 새 가설이 생기면 **즉시 5단계 장부로 복귀**(순회 중단). h까지 전부 수행 완료·보류인데
생존 가설 0이면 재순회 금지 → 9단계 에스컬레이션.

**a) 가정 감사 (가장 수확 많음)** — 가정 생성 절차: 3단계 인과 사슬의 **링크마다 1행씩** "이
링크는 실제로 작동한다"를 등록(사슬 7링크면 최소 7행) + 환경 가정 3종 상시 추가(①실행 중인
코드=방금 저장한 코드다(HMR·캐시) ②읽는 설정/파일 경로=내 생각의 그 경로다 ③입력이 의도대로
전달됐다) + 부정 능력 가정("이 기능의 API는 존재하지 않는다" — grep으로 검증, wasm-bridge
320+ 실사례):
```
| 가정 | 실측했나? | 검증 방법 | 결과(참/거짓/보류) |
| 이벤트가 리스너에 도달한다 | ✗ | 리스너 첫 줄 로그 | |
| 이 env 파일이 읽힌다 | ✗ | 로드 지점 로그/경로 출력 | |
| 핸들러가 매 프레임 돈다 | ✗ | 내부 카운터 | |
```
가정당 검증 실험 1개·15분 상한 — 상한 초과는 "보류"로 표시하고 리스크로 기록 후 진행.
**a의 종료 조건 = 전 가정의 판정(참/거짓/보류) 완료** (전부 '참'이 아니라 전부 '판정').
거짓 판명 = 즉시 5단계 장부에 신규 가설로 등록. → 실사례: "핸들러가 돈다" 실측 → RAF가
자동화에서 안 돎 / "loadEnv가 이 폴더를 읽는다" → cwd가 부모 폴더.

**b) 층 내리기** — 미스터리한 API의 소스를 직접 읽는다(래퍼 wasm-bridge.ts, vite proxy 구현,
상류 호출부 grep). 문서가 아니라 구현이 계약이다.

**c) 유사 기제 이식 (analogy)** — "같은 목표를 이 코드베이스 다른 곳은 어떻게 이루나?"
검색 레시피: ①목표에 쓰일 법한 API명·속성명·상수명(vertRelTo, offset, borderFill)으로 전체
grep ②히트 중 CLAUDE.md "검증 완료" 절·verify 하네스가 참조하는 파일을 우선 신뢰 ③그 호출
형태(파라미터·순서·전후 처리)를 그대로 복제해 실험 E로 등록. → 실사례: 표 위치의 답(Paper
절대 오프셋)은 exportCore의 검증된 HWPX 매핑에서 왔다.

**d) 상류 지식 발굴** — 업스트림의 테스트가 계약의 문서다. tests/를 증상 키워드로 grep. 이슈·
체인지로그도. → 실사례: #1491 성능 계약은 소스 계약 테스트에 적혀 있었다.

**d′) 참조 구현 오라클** — 미문서 포맷·프로토콜은 "이미 그것을 할 줄 아는 구현"에 같은 작업을
시키고 출력을 diff해 계약을 읽는다. → 실사례: 누름틀 필드의 HWPX 구조는 rhwp에
insertClickHereField 시키고 exportHwpx 결과를 XML diff — 문서 0줄로 계약 확정(hp:pic 동일).

**e) 반전 질문** — "이 증상이 나오려면 무엇이 참이어야 하는가?"를 거꾸로 나열. → 실사례:
"규칙 있고 클래스 붙었는데 176px"가 나오려면? ①규칙 미매칭(기각) ②다른 규칙이 이김(기각:
인라인 width:0도 실패) ③레이아웃 제약이 width 무시 → min-content 방향 → 자식의 인라인
display:flex 발견.

**f) 다중 원인 의심** — 수리가 증상의 **일부만** 설명하면 원인이 2개 이상이다. **증상이 전
방향·전 방법에서 균일하게 무반응이면** 개별 버그보다 상류 게이트 또는 다중 원인을 의심.
원인 하나 찾을 때마다 남은 증상 목록에서 지워가며 0이 될 때까지. → 실사례: 표 행높이 회귀 =
Tailwind 충돌+병합 앵커 **2중** / 표 리사이즈 전 방향 무반응 = 캐시 조건+pageHint 비교 버그+
핸들 이동 소비 **3중**.

**g) 시간축·환경축 차등** — 언제부터(git log -S, bisect) / 어디서만(자동화 vs 실사용, dev vs
빌드, 로컬 vs Codespaces, 캐시·HMR 잔재 — 스택의 ?t= 타임스탬프로 과거 로드 식별).

**h) 문제 재정의** — 고치는 대신 그 요구를 없앨 수 있나? (H1: "현재 모델로 목표 상태를 직렬화할
수 있는가" — 안 되면 버그 수정이 아니라 구조 변경이 답.) → 실사례: 생성→이동을 고치는 대신
"생성 시점에 floating으로 태어나게"(createTableEx).

## 8단계. 수리 규율

- **근인 vs 근본** — 어느 층에 패치할지 명시적으로 결정하고 이유를 기록. 증상 층 패치(우회)가
  정당한 경우도 있다(상류 소유 코드) — 단 "우회임"을 주석·traps에 남긴다.
- **전 증상 회계**: 수리 전/후 증상·수치 목록을 나란히 적어 설명 안 된 항목이 기계적으로
  드러나게 한다. 잔여 증상 = 5단계 장부에 "신규 관찰"로 등록하고 **7f(다중 원인)부터 수사
  재개**. "고쳐진 것 같음" 종료 금지.
- **종료 기준**: 결정적 버그(5/5 재현)는 E0 1회 통과. 간헐 버그는 0단계 재현율 표본의 2배
  이상(최소 10회) 연속 무재현.
- **잔차 보정 루프**: 큰 수리 후 작은 오차는 측정→보정→재측정 수렴 루프로(1mm 바깥여백 실사례).
  보정 3회 내 오차가 단조 감소하지 않으면 수렴 실패 — 수리 자체를 재검토(5단계 복귀).
- **회귀 방지 판정표** — 하나라도 참이면 **반드시** 테스트/게이트로 계약을 박는다: ①수리에
  수치 계약이 있다(오차 한계·개수·크기) ②이번 버그가 회귀였다 ③기존 하네스가 같은 대상을
  검사한다. 셋 다 아니면 traps.md 3줄로 대체 가능. (실사례: #1491 계약을 게이트 검사로 강화.)

## 9단계. 종료와 인수인계

- **해결 시**: traps.md에 3줄(증상→원인→해법) + 큰 건은 rhwp-adoption.md에 서사. 기각 가설은
  실험 2개 이상을 소모했거나 다음 사람이 첫 가설로 세울 법한(부록 A에 걸리는) 것이면 남긴다.
- **에스컬레이션 (7단계 예산 소진 시에만)**: `debug-*.md` 장부 파일 경로를 알려주고, 보고
  메시지에 **장부 표 전문 + 인과 사슬 + E번호별 결과**를 그대로 붙인다(요약본만 넘기기 금지) —
  다음 모델이 0이 아니라 당신이 멈춘 지점에서 시작하게.
