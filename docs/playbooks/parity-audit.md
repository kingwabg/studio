# 패리티 감사 (parity-audit) — "X와 동일하게 만들어줘"를 실행 절차로

목적: "경쟁 제품(한컴 등)과 동일하게" 류 요청을 통짜 분석·감 없이 **측정 가능한 갭 목록 →
순차 수정**으로 바꾸는 절차. 유래: 2026-07-12 표 기능 요청 — 사용자가 제안한 4단계 리버스
파이프라인(정적 분석→동적 조작→네트워크→재구성)을 우리 환경·법 경계에 맞게 규칙화.
첫 적용 = 표 영역 (체크리스트 `docs/parity/table.md`, 1호 갭 수정 = 셀 Ctrl+A 확장 선택).

## 언제 쓰나
- 요청이 기능 1개가 아니라 **영역**("표 전부", "이미지 편집 전부")을 가리킬 때.
- 단일 버그 신고면 이 문서가 아니라 traps grep → debugging.md 경로.

## 파이프라인 (4단계 — ①②는 병렬 가능)

### ① 코드 인벤토리 — 우리가 뭘 가졌나 (서브에이전트 위임, 읽기 전용)
- 대상 영역의 커맨드·핸들러·키맵·메뉴 진입점을 전수 나열.
- 산출: `기능 | 진입점(커맨드 id/키/함수) | 상태[OK/불완전/스텁] | 근거(파일:줄)`.
- 스텁 단서 grep: `TODO`·`미구현`·`console.warn`·무조건 false 반환·죽은 shortcutLabel.

### ② 런타임 실측 — 코드에 있어도 실제로 되나
- browser-drive.md 레시피로 실구동. ①에서 OK로 나온 항목도 **대표 시나리오 1개씩 실측**
  — "코드는 있는데 동작이 틀린" 갭(예: Ctrl+A가 문서 전체로 튐)은 여기서만 잡힌다.
- 산출: ①의 표에 두 열을 덧붙여 같은 표를 확장 — `실측[일치/갭/애매] | 근거(호출·수치)`.

### ③ 레퍼런스 행동 관찰 — 정답이 뭔가 (블랙박스만)
- ②에서 갭/애매로 판정된 항목**만** 그때그때 확인 (전 항목 일괄 분석 금지).
- 정본 서열과 **관찰 주체**:
  - 1차 = 실제 한컴 한글(데스크톱) — **사용자가 관찰한다.** 에이전트는 "조건 → 기대 동작"
    질문 목록을 만들어 요청하고, 답이 오기 전엔 그 항목을 **'기대 동작 미확정'**으로 표기한
    채 다른 항목을 진행 (verify.md의 '사용자 확인 필요' 프로토콜 준용 — 기억으로 지어내기 금지).
  - 보조 = 웹한글(webhwp)·HWPX 표준 문서 — 에이전트가 아래 허용 범위에서 직접 확인 가능.
- **허용/금지의 조작적 정의** (도구 수준):
  - 허용: 정상 계정으로 **수동 조작**해 화면에 보이는 결과를 관찰 — 키 입력·클릭 후 상태를
    스크린샷/렌더로 확인. read_page류 접근성 읽기는 **화면 텍스트 확인까지만**.
  - 금지 (전부, 예외 없음):
    - 상용 제품(웹·데스크톱 불문)의 코드·번들·DOM 구조·내부 상태(전역 객체·이벤트 핸들러)
      추출·기록·복제.
    - 네트워크 페이로드 수집·API 규격 리버스.
    - 레퍼런스 서비스에 대한 **스크립트 자동화 접근(규모 불문 — 관찰은 수동 조작으로)** ·
      로그인 장벽/봇 감지 우회 (정상 계정으로 수동 로그인해 써보는 것은 허용).
    - 설치된 상용 프로그램(한컴 한글 등)의 디컴파일·바이너리/DLL 분석·내장 리소스
      (서식 템플릿·문자열 등) 추출 (EULA).
  - 근거: **①법/약관 — 이것만으로 무조건 금지(유용성과 무관).** ②덤으로 우리 구조상 어차피
    무용 — 우리 표는 WASM HWP 문서 모델이라 남의 구현 세부는 이식 불가. 필요한 건 행동
    명세뿐이고, 동작·아이디어는 저작권 대상이 아니다 (제품을 써보고 배우는 것과 동일).
  - **회색지대**(목록에 없는데 법/약관이 걸릴 것 같은 행동) = 하지 말고 사용자에게 질문
    (agent-protocol §3 에스컬레이션 준용).

### ④ 갭 수정 루프
- **갭 표 생성 규칙**: ①②확장 표에서 실측≠일치 행만 추리고 ③의 행동 명세를 붙인다 —
  `기능 | 우리 상태 | 기대 동작(출처·날짜 / '미확정') | 점수`.
- **우선순위 점수 = 빈도 × 깨짐** (척도 고정 — H6):
  - 빈도: 3=거의 매 문서 / 2=주 단위 / 1=드묾.
  - 깨짐: 3=오동작(틀린 결과·데이터 위험) / 2=불완전·비일관 / 1=기능 부재.
    (오동작 > 부재 — 있는데 틀린 것이 없는 것보다 신뢰를 더 깎는다.)
  - 컷: **6+ = 즉시 · 3~4 = 다음 배치 · 1~2 = 백로그.**
- **실행**: 한 갭 = 한 커밋 + 실측 검증(verify.md 게이트). 브랜치·머지 리듬·구역 분할은
  **agent-protocol §1.5를 따른다** (특히 표/입력 해석 = A구역 핫스팟, 동시 1세션 — 착수 전
  `git branch -r`로 타 세션 확인; 머지는 배치 검토 때 몰아서).
